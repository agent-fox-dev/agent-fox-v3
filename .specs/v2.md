# V2 ROADMAP planning

## 1. Product Overview

agent-fox v2 transforms from a spec executor into a **goal-driven software development partner** — an autonomous system that takes high-level product intent and shepherds it all the way to shipped, maintained, documented code, growing more capable and project-specific with every session.

Where v1 asks "what are the tasks?", v2 asks "what are you trying to build, and why?" It reasons backward from goals to specs, forward from specs to code, sideways to maintenance and health — all while accumulating institutional knowledge that makes every future session faster, cheaper, and more accurate than the last.

**For whom:** Small engineering teams (1-5 people) building software products who want to move at the speed of a large team without the overhead. Also: solo founders, solo engineers, and research teams.

**The flywheel:** Every session makes the fox smarter about your specific project. After three months, it knows your codebase architecture, your team's patterns, your recurring failure modes, and your conventions better than any new hire ever would. After a year, it is irreplaceable institutional memory made executable.

## 2. Goals and Non-Goals

### Goals

- **Reduce the human authoring burden** — teams provide goals and intent; the fox generates and maintains the spec tree
- **Make the knowledge flywheel visible and queryable** — the accumulated project knowledge is accessible to humans, not buried in JSONL
- **Enable autonomous overnight operation** — teams return each morning to reviewed, merged, tested maintenance PRs the fox opened while they slept
- **Deepen multi-agent coordination** — parallel agents don't just share state; they share discoveries and adapt in real-time
- **Bring the fox into the IDE** — ambient project intelligence while humans code, not just when they invoke the CLI
- **Remain radically interruptible and auditable** — every decision the fox makes is logged, explainable, and reversible

### Non-Goals

- **Not a general-purpose coding assistant** — agent-fox v2 is a project-aware orchestrator, not a replacement for Cursor or Copilot at the autocomplete level
- **Not a hosted SaaS** — v2 continues to run locally/self-hosted; the team's code never leaves their infrastructure (except what they explicitly push to GitHub)
- **Not an AGI planner** — the fox reasons about code, not about business strategy; product decisions remain with humans
- **Not a CI/CD replacement** — the fox produces code for human review; it does not replace your pipeline

## 3. User Personas

| Persona | Who They Are | What v2 Does for Them |
|---------|-------------|----------------------|
| **The Founder-Engineer** | Solo founder building a product, writing all code alone | The fox is the 2nd, 3rd, and 4th engineer they can't afford to hire. It maintains health, documents decisions, onboards the first hire, and ships features overnight. |
| **The Small Team Lead** | 2-4 person team, everyone writes code and reviews code | The fox eliminates the "3am maintenance debt" and gives the team a shared institutional memory that persists through turnover. |
| **The Researcher** | Academic or R&D engineer running experiments, not building products | The fox manages iterative experiment specs, extracts learnings, and maintains reproducibility so the researcher can focus on thinking, not plumbing. |
| **The New Hire** | Just joined a 3-year-old codebase | The fox gives them a personalized onboarding package synthesized from project history — not stale wiki docs, but ground-truth knowledge. |


## 4. The Nine Tails in Detail

---

### Tail 1: The Fox Ball — Semantic Knowledge Oracle

> *The hoshi no tama — the fox's magical jewel — contains its accumulated power. Whoever holds it holds leverage over the fox.*

**The problem today:** agent-fox v1 stores knowledge as flat JSONL facts. The facts are useful for future prompts but they're not queryable by humans. You can't ask "why did we choose Postgres over SQLite?" or "which sessions touched the auth module?" The institutional knowledge exists but is locked inside the machine.

**The v2 solution:** The Fox Ball is a queryable knowledge graph built from everything agent-fox has ever seen about your project: spec history, memory facts, ADRs, session outcomes, test failures, commit messages, code structure, and the temporal relationships between all of them.

**What it looks like:**

```bash
# Query the knowledge graph in natural language
agent-fox ask "why do we use Redis for session storage?"
agent-fox ask "which specs touched the user authentication module?"
agent-fox ask "what has broken the payments flow historically?"
agent-fox ask "what would be the impact of removing the legacy API endpoint?"
```

**How `agent-fox ask` works:** The command is a single-round retrieval-augmented generation (RAG) pipeline. The query is embedded, semantically similar facts are retrieved from DuckDB via vector search, and the retrieved context is passed to the STANDARD_MODEL (Sonnet) in a single API call to synthesize a grounded answer. The response includes confidence level and source citations. No streaming — the answer is returned as a complete block. If the knowledge store is empty (fresh project with no sessions), the command prints a clear message explaining that no knowledge has been accumulated yet and suggests running a few coding sessions first.

**What it produces:**

```
The fox ball illuminates:

  Redis was chosen in spec 03_session_store (session 4, 2025-11-03) because
  the team needed pub/sub capabilities for real-time notifications (ADR-04).
  SQLite was initially proposed but rejected — see the decision rationale
  in session 03_session_store/4. The choice was reaffirmed in spec
  15_platform_integration when Sidekiq was added.

  Confidence: high (3 independent sources)
  Sources: ADR-04, memory fact e7f2a..., session 03_session_store/4
```

The Fox Ball is the foundation every other tail builds on. It's what makes v2 a knowledge institution, not just a task runner.

**How it's built:**
- Every session continues to extract structured facts (v1 behavior)
- Facts are dual-written: appended to the existing JSONL store (backward compatibility, human readability) and indexed into DuckDB with embeddings (queryability)
- Fact supersession is tracked explicitly via the existing `supersedes` field, mirrored into DuckDB's `superseded_by` column
- Contradiction and confirmation between facts are detected at query time via semantic similarity — when retrieved facts conflict, the query model flags the contradiction in its response. No automatic detection at write time (too expensive for the marginal benefit)
- ADRs, git history, spec history, and test outcomes are ingested as first-class knowledge sources
- The knowledge store lives at `.agent-fox/knowledge.duckdb`, gitignored alongside the project

### Tail 2: Time Vision — Temporal Reasoning and Predictive Memory

> *An old fox doesn't just remember what happened. It remembers why it happened, and what came before, and what came after.*

**The problem today:** v1 memory extracts facts that are timestamped but not causally linked. "User.email became nullable" is a fact, but "User.email became nullable because of spec 07 which was triggered by issue #34 about third-party OAuth support, which caused spec 09 tests to break three weeks later" — that's a causal chain, and v1 can't represent it.

**The v2 solution:** Memory as a temporal causal graph. Every fact has:
- **provenance**: which session, which spec, which commit produced it
- **cause**: what event or decision led to this fact
- **effects**: what downstream facts or failures this fact caused
- **timeline**: when it became true, and whether it was superseded

This enables temporal queries:

```bash
agent-fox ask "what happened the last time we changed the auth module?"
# Returns a timeline: spec 05 changed auth → spec 09 tests broke →
# memory fact M:4f3a noted the fragility → spec 12 added integration tests

agent-fox ask "what has failed repeatedly in the last 6 months?"
# Returns: payments → auth edge case, User model changes, Redis timeout handling
```

**Predictive signals:** The temporal graph enables the fox to notice patterns:

```
Temporal pattern detected:

  The last 3 times specs touched src/auth/ (specs 05, 09, 17),
  tests in test_payments.py failed within the same run.

  Spec 24_billing currently proposes changes to src/auth/session.py.
  Recommend: pre-run test_payments.py before executing 24_billing/3.
```

Time Vision turns historical data into actionable foresight. The fox gets smarter about your specific failure modes, not just general software development patterns.

## 5. v2 Knowledge Store: DuckDB

Settled decision for the persistence layer backing Tail 1 (Fox Ball) and Tail 2 (Time Vision).

---

### What was evaluated

| Option | Verdict |
|---|---|
| AliSQL | Eliminated — requires a running MySQL server. Incompatible with the fox's embedded, self-contained model. |
| ChromaDB | Eliminated — HNSW advantage only materialises at >100k vectors. Fox datasets are hundreds to low thousands of facts. Higher dependency weight for no practical gain. |
| sqlite-vec | Strong candidate — minimal dependency (SQLite extension), exact KNN sufficient at fox scale, SQL joins with structured data. |
| **DuckDB** | **Selected** — covers both tails in one file; VSS extension adds vector search alongside native columnar analytics. |

sqlite-vec was the front-runner until DuckDB's VSS extension (HNSW vector
search, `INSTALL vss; LOAD vss;`) was factored in. Since DuckDB was already
required for Time Vision's analytical queries, adding VSS costs nothing new and
eliminates the need for a second database.

---

### Why DuckDB

**One file, one connection, full SQL across both tails.**

`.agent-fox/knowledge.duckdb` holds everything: embedded fact vectors, temporal
session records, causal links, and pattern history. Cross-tail queries run in a
single SQL statement with no application-layer joins:

```sql
-- "Find facts semantically close to this query, but only from specs
--  that touched src/auth/ in the last 6 months"
SELECT f.content, f.spec_name, v.distance
FROM memory_facts f
JOIN (
    SELECT id, distance
    FROM memory_embeddings
    WHERE embedding <-> ? < 0.4
    ORDER BY distance LIMIT 20
) v ON f.id = v.id
WHERE f.spec_name IN (
    SELECT spec_name FROM session_outcomes
    WHERE touched_path LIKE 'src/auth/%'
      AND created_at > NOW() - INTERVAL 6 MONTHS
)
ORDER BY v.distance;
```

This is the kind of query that makes Tail 1 genuinely useful rather than just
a search box. It is only possible because vector results and structured session
metadata live in the same database.

**Fits the fox's embedded contract.** DuckDB is a `pip install duckdb` — no
server, no port, no auth, no startup ordering. Data is a single file inside
`.agent-fox/`, gitignored alongside the rest of the project.

**Platform compatibility.** DuckDB and its VSS extension support macOS (ARM and
x86), Linux (x86_64, aarch64), and Windows. This covers the fox's supported
platforms. The VSS extension is installed automatically on first use via
`INSTALL vss;` and loaded on subsequent connections via `LOAD vss;`.

**Columnar OLAP is native.** Time Vision's temporal pattern queries —
"the last N times a session touched this path", "which specs produced repeated
failures", windowed aggregations over session timelines — are exactly the
workload DuckDB is built for. InnoDB-style row storage would struggle here;
DuckDB handles it without configuration.

---

### Schema sketch

```sql
-- Schema versioning for forward-compatible migrations
CREATE TABLE schema_version (
    version     INTEGER PRIMARY KEY,
    applied_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    description TEXT
);

-- Tail 1: knowledge facts with embeddings
CREATE TABLE memory_facts (
    id            UUID PRIMARY KEY,
    content       TEXT NOT NULL,
    category      TEXT,             -- gotcha, pattern, decision, etc.
    spec_name     TEXT,
    session_id    TEXT,
    commit_sha    TEXT,             -- provenance: which commit produced this fact
    confidence    TEXT DEFAULT 'high',
    created_at    TIMESTAMP,
    superseded_by UUID              -- causal link: this fact was replaced by
);

CREATE TABLE memory_embeddings (
    id        UUID PRIMARY KEY REFERENCES memory_facts(id),
    embedding FLOAT[1024]  -- dimension is model-derived (1024 for voyage-3)
);
CREATE INDEX ON memory_embeddings USING HNSW (embedding);

-- Session outcomes (always-on, not gated by --debug)
CREATE TABLE session_outcomes (
    id           UUID PRIMARY KEY,
    spec_name    TEXT,
    task_group   TEXT,
    node_id      TEXT,
    touched_path TEXT,  -- one row per path touched (normalised)
    status       TEXT,  -- completed / failed / timeout
    input_tokens  INTEGER,
    output_tokens INTEGER,
    duration_ms   INTEGER,
    created_at   TIMESTAMP
);

-- Temporal causal graph
CREATE TABLE fact_causes (
    cause_id  UUID REFERENCES memory_facts(id),
    effect_id UUID REFERENCES memory_facts(id),
    PRIMARY KEY (cause_id, effect_id)
);
```

The `superseded_by` column and `fact_causes` table together form the causal
chain that Time Vision reasons over. The HNSW index on `memory_embeddings` is
what the Fox Ball queries for semantic search.

The `schema_version` table enables forward-compatible migrations: on startup,
the fox checks the current schema version and applies any pending migrations.
Migrations are forward-only (no rollback) and shipped as numbered Python
functions in the codebase.

---

### Embedding model

DuckDB stores and searches vectors; it does not generate them. Embeddings are
produced at write time by the fox before inserting a fact.

**Default: Anthropic voyage-3 API** — consistent with the existing Anthropic API
relationship, no new vendor, reuses the existing API key. The voyage-3 model
produces 1024-dimensional vectors, which is what the schema uses. If the API
key lacks embedding permissions, the fox logs a warning and skips embedding
generation (facts are still written to DuckDB and JSONL without vectors, and
can be backfilled later).

The embedding dimensionality is model-derived, not hardcoded. If a future
configuration option switches to a different model (e.g., a local
`sentence-transformers` model with 384 dimensions), the schema adapts
accordingly. The `FLOAT[N]` array size is set during initial schema creation
based on the configured model's output dimensionality.

Local model support via `sentence-transformers` is a future option — offline,
no API cost, but slower on first run and adds a heavy dependency. Not included
in the initial v2 implementation.

### Audit log integration

The `code --debug` command writes a full audit trail to a timestamped JSONL
file in `.agent-fox/`: every SDK message (assistant turns, tool use/results,
thinking blocks) plus all DEBUG-level log lines. The question is how much of
this should flow into DuckDB.

The answer is: structured signals always, raw content never, and the
`--debug` gate needs to be split.

#### Three-layer model

| Layer | Destination | When |
|---|---|---|
| Session start/end metadata | DuckDB `session_outcomes` | **Always** — unconditional |
| Tool call and error signals | DuckDB `tool_calls`, `tool_errors` | `--debug` only |
| Raw message content | JSONL audit file | `--debug` only |

#### What goes where and why

**Always → DuckDB `session_outcomes`:** Session start/end events are cheap,
structured, and exactly what Time Vision needs for temporal pattern queries.
They should not be gated behind `--debug`. Currently they only exist in the
audit file when `--debug` is passed — that is a bug in the v1 design that v2
must fix.

**`--debug` → DuckDB `tool_calls` / `tool_errors`:** The message stream
contains `ToolUseBlock` and `ToolResultBlock` events with structured data
(tool name, inputs, error flag). Extracting these into DuckDB enables
analytical queries across sessions: which tools does the agent reach for most
in this project, which tool calls keep failing, how does tool usage correlate
with session success. This is worth indexing but only when debug mode is
active since parsing the full message stream has cost.

**`--debug` → JSONL only:** Raw assistant text, thinking blocks, and tool
result content (bash output, file reads) can be gigabytes per session. They
are useful for forensic replay of a specific session, not for aggregate
analysis. DuckDB is the wrong store for them.

#### Schema additions for debug-only signals

```sql
-- Extracted from ToolUseBlock events (debug-only)
CREATE TABLE tool_calls (
    id          UUID PRIMARY KEY,
    session_id  TEXT,
    node_id     TEXT,
    tool_name   TEXT,
    called_at   TIMESTAMP
);

-- Extracted from ToolResultBlock events where is_error = true (debug-only)
CREATE TABLE tool_errors (
    id          UUID PRIMARY KEY,
    session_id  TEXT,
    node_id     TEXT,
    tool_name   TEXT,
    failed_at   TIMESTAMP
);
```

#### The SessionSink abstraction

The current `AuditWriter` writes to JSONL and nothing else. v2 introduces a
`SessionSink` protocol with two implementations:

- `JsonlSink` — current behaviour, debug-only, raw full-fidelity archive
- `DuckdbSink` — new, always-on for session outcomes, optional for tool signals

Both can be active simultaneously. The orchestrator attaches sinks based on
the `--debug` flag:

- **Always:** `DuckdbSink` is attached (session outcome methods active; tool
  signal methods are no-ops unless debug is enabled)
- **With `--debug`:** `JsonlSink` is also attached, and `DuckdbSink` enables
  tool signal recording

The session runner calls sink methods without knowing or caring which sinks
are active. The orchestrator's sink attachment logic is the single place
where the debug/always decision is made.

---

### JSONL dual-write strategy

During the v2 transition, memory facts are dual-written:

1. **JSONL** (`.agent-fox/memory.jsonl`) — continues as the human-readable,
   backward-compatible store. The existing `docs/memory.md` rendering pipeline
   continues to work unchanged.
2. **DuckDB** (`.agent-fox/knowledge.duckdb`) — adds queryability, embeddings,
   and cross-session analytics.

The JSONL store remains the source of truth for fact content. DuckDB is a
queryable index that can be rebuilt from JSONL + git history if corrupted.
This means JSONL deletion is a data-loss event, but DuckDB deletion is a
recoverable inconvenience.

In a future version, JSONL may be deprecated once DuckDB has proven stable
across diverse projects. That deprecation is explicitly out of scope for v2.

---

### Resilience

**DuckDB corruption:** If `knowledge.duckdb` is corrupted or unreadable, the
fox logs an error, continues operating without the knowledge store, and warns
the user to re-initialize. The `agent-fox ask` command returns a clear error
message. Coding sessions are unaffected — they do not depend on DuckDB. This
mirrors the existing JSONL corruption handling: graceful degradation, never a
hard crash.

**Re-initialization:** A future `agent-fox init --rebuild-knowledge` command
can rebuild DuckDB from the JSONL fact store and git history. This is out of
scope for the initial v2 implementation but the dual-write strategy makes it
possible.

---

### Operational notes

- DuckDB's WAL provides crash safety; no explicit transaction management needed
  for append-only fact writes.
- The VSS extension must be installed once: `INSTALL vss;`. Subsequent
  connections just `LOAD vss;`. This is handled transparently in the fox's
  database initialisation path — the first connection installs and loads, all
  subsequent connections only load.
- For parallel sessions (spec 06), DuckDB supports multiple readers but only
  one writer. Fact writes happen post-harvest (serial by design), so no
  write contention arises from the existing parallel execution model. Session
  outcome writes from parallel workers are serialized through the existing
  state lock.

## 6. Implementation Phasing

The v2 knowledge system is implemented as three sequential specs:

| Spec | Name | Scope |
|------|------|-------|
| **23** | `duckdb_knowledge_store` | DuckDB infrastructure, schema, connection management, SessionSink protocol, always-on session outcome recording |
| **24** | `fox_ball` | Fact indexing + embeddings, vector search, knowledge source ingestion, `agent-fox ask` command |
| **25** | `time_vision` | Causal graph, provenance tracking, temporal queries, predictive pattern detection |

Dependencies flow cleanly: 23 → 24 → 25. Each spec is a cohesive vertical
slice that can be implemented, tested, and shipped independently.
