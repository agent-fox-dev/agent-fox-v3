# Requirements Document

## Introduction

This specification defines the simplification of the `agent-fox` configuration
template and the creation of comprehensive configuration reference
documentation. The goal is to reduce the generated `config.toml` from ~190
lines to ~40 lines of essential, well-documented options with quality-first
defaults, while providing a full reference document for advanced users.

## Glossary

- **Promoted field**: A config field rendered as active (uncommented) in the
  generated template, with its value set rather than commented out.
- **Visible section**: A TOML section that appears in the simplified template.
  Only essential sections are visible; others are omitted entirely.
- **Hidden section**: A TOML section omitted from the simplified template but
  fully supported when manually added by the user.
- **Quality gate**: A shell command executed after each coder session to verify
  code quality (e.g., `make check`, `npm test`).
- **Template**: The `config.toml` content generated by `agent-fox init`.
- **Config reference**: The `docs/config-reference.md` document listing every
  available configuration option.
- **Field description**: The `##` comment above a config field explaining what
  it controls.

## Requirements

### Requirement 1: Simplified Template Generation

**User Story:** As a user running `agent-fox init`, I want a short, clear
config file so that I can understand and customize the essential settings
without being overwhelmed by options I don't need.

#### Acceptance Criteria

1. [68-REQ-1.1] WHEN `generate_default_config()` is called, THE system SHALL
   produce a template containing only visible sections: `orchestrator`,
   `models`, `archetypes` (with `instances` and `thinking` subsections), and
   `security`.

2. [68-REQ-1.2] WHEN `generate_default_config()` is called, THE system SHALL
   omit all hidden sections: `routing`, `theme`, `platform`, `knowledge`,
   `pricing`, `planning`, `blocking`, `hooks`, and `night_shift`.

3. [68-REQ-1.3] THE template SHALL contain a footer comment referencing the
   full configuration documentation: `docs/config-reference.md`.

4. [68-REQ-1.4] THE template SHALL contain no more than 60 lines (including
   blank lines and comments).

5. [68-REQ-1.5] WHEN a hidden section is manually added to an existing
   `config.toml`, THE system SHALL load and validate it identically to visible
   sections.

#### Edge Cases

1. [68-REQ-1.E1] IF a user's existing config contains hidden sections and
   `merge_existing_config()` is called, THEN THE system SHALL preserve those
   sections without modification.

2. [68-REQ-1.E2] IF `generate_default_config()` is called, THEN THE system
   SHALL produce valid TOML that parses without errors.

### Requirement 2: Quality-Biased Promoted Defaults

**User Story:** As a user, I want the default config to prioritize code quality
and verification so that my projects start with thorough testing and review
enabled out of the box.

#### Acceptance Criteria

1. [68-REQ-2.1] THE template SHALL render `orchestrator.quality_gate` as a
   promoted field with the value `"make check"`.

2. [68-REQ-2.2] THE template SHALL render `archetypes.instances.verifier` as a
   promoted field with the value `2`.

3. [68-REQ-2.3] THE template SHALL render all quality-related archetype toggles
   (`skeptic`, `verifier`, `oracle`, `auditor`) as promoted fields with the
   value `true`.

4. [68-REQ-2.4] THE template SHALL render `orchestrator.max_budget_usd` as a
   promoted field with the value `5.0`.

5. [68-REQ-2.5] THE template SHALL render `models.coding` as a promoted field
   with the value `"ADVANCED"`.

6. [68-REQ-2.6] THE code default for `ArchetypeInstancesConfig.verifier` SHALL
   be `2`.

#### Edge Cases

1. [68-REQ-2.E1] IF `archetypes.coder` is set to `false` in config, THEN THE
   system SHALL override it to `true` and log a warning (existing behavior,
   preserved).

### Requirement 3: Improved Field Descriptions

**User Story:** As a user reading `config.toml`, I want each setting to have a
clear description explaining what it does and why I might change it, so that I
can configure the tool without reading source code.

#### Acceptance Criteria

1. [68-REQ-3.1] WHEN the template is generated, THE system SHALL render a
   description comment (prefixed with `##`) above every promoted field.

2. [68-REQ-3.2] THE description for each promoted field SHALL explain the
   field's purpose in plain language, not merely restate the field name.

3. [68-REQ-3.3] THE description for bounded fields SHALL include the valid
   range (e.g., `1-8`, `>=0`).

#### Edge Cases

1. [68-REQ-3.E1] IF a field lacks a description in the schema metadata, THEN
   THE system SHALL use the description from `_DEFAULT_DESCRIPTIONS` or
   generate a readable fallback from the field name.

### Requirement 4: Config Reference Documentation

**User Story:** As an advanced user, I want a comprehensive reference document
listing every configuration option so that I can discover and use hidden
sections without reading source code.

#### Acceptance Criteria

1. [68-REQ-4.1] THE system SHALL include a `docs/config-reference.md` file in
   the repository documenting every config section and field.

2. [68-REQ-4.2] WHEN a field is documented in `config-reference.md`, THE
   document SHALL include the field's type, default value, valid range (if
   bounded), and a plain-language description.

3. [68-REQ-4.3] THE document SHALL include TOML examples for complex field
   types (inline tables, nested sub-tables like `[archetypes.thinking.coder]`).

4. [68-REQ-4.4] THE document SHALL be organized by section with a table of
   contents linking to each section.

#### Edge Cases

1. [68-REQ-4.E1] IF the config schema is updated in future specs, THEN THE
   `docs/config-reference.md` SHALL be treated as a living document that
   requires manual updates (not auto-generated).

### Requirement 5: Re-initialization Behavior

**User Story:** As a user re-running `agent-fox init`, I want the output to be
consistent with the simplified format so that re-initialization does not
reintroduce the verbose template.

#### Acceptance Criteria

1. [68-REQ-5.1] WHEN `agent-fox init` is run on an existing project (merge
   mode), THE system SHALL preserve all active user values and produce output
   consistent with the simplified format.

2. [68-REQ-5.2] WHEN `agent-fox init --force` is run, THE system SHALL
   produce the simplified template, discarding the existing config.

3. [68-REQ-5.3] WHEN `merge_existing_config()` is called, THE system SHALL NOT
   add commented-out entries for hidden sections that were not already present
   in the existing config.

#### Edge Cases

1. [68-REQ-5.E1] IF the existing config contains deprecated fields, THEN THE
   system SHALL mark them with `# DEPRECATED:` comments (existing behavior,
   preserved).

2. [68-REQ-5.E2] IF the existing config is empty or whitespace-only, THEN THE
   system SHALL produce the simplified template (not the full verbose template).

### Requirement 6: Template Footer

**User Story:** As a user, I want the config file to tell me where to find
documentation for advanced options so that I know hidden sections exist.

#### Acceptance Criteria

1. [68-REQ-6.1] THE template SHALL end with a blank line followed by a comment
   line: `## For all configuration options, see docs/config-reference.md`.

#### Edge Cases

1. [68-REQ-6.E1] IF `merge_existing_config()` is called on a config that
   already contains the footer comment, THEN THE system SHALL NOT duplicate the
   footer.
