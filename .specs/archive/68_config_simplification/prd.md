# PRD: Config Simplification

## Problem

The current `config.toml` generated by `agent-fox init` is overwhelming. It
dumps every possible configuration option (~190 lines, 13+ sections), with most
options commented out. The comments are terse ("default: true") without
explaining what the parameter actually controls or why a user would change it.
There is no guidance on which settings matter for code quality and verification.

Users must read source code to understand what most options do.

## Goal

Create a simpler, opinionated config template that:

1. **Shows only essential options** — the 80/20 of settings that actually matter
   for day-to-day use.
2. **Provides clear, actionable descriptions** — explain *what the setting
   does* and *why you'd change it*, not just the default value.
3. **Ships with quality-first defaults** — bias toward thorough testing,
   verification, and code quality out of the box.
4. **Points users to full documentation** — a footer referencing
   `docs/config-reference.md` for advanced options.
5. **Creates comprehensive reference docs** — a full `docs/config-reference.md`
   documenting every available option with descriptions, defaults, bounds, and
   examples.

## Visible Sections (Essential)

Only these sections appear in the simplified template:

- **`[orchestrator]`**: `parallel`, `max_budget_usd`, `quality_gate`
- **`[models]`**: `coding`
- **`[archetypes]`**: toggle flags (skeptic, verifier, oracle, auditor) +
  `instances.verifier`
- **`[archetypes.thinking]`**: coder, skeptic, verifier thinking config
- **`[security]`**: `bash_allowlist_extend`

Everything else (routing, theme, platform, knowledge, pricing, planning,
blocking, hooks, night_shift) is omitted from the template — users add those
sections manually when needed, guided by the reference doc.

## Quality-Biased Defaults

The template ships with these values **uncommented and active**:

| Field | Value | Rationale |
|-------|-------|-----------|
| `orchestrator.parallel` | `2` | Sensible concurrency |
| `orchestrator.max_budget_usd` | `5.0` | Reasonable per-session cap |
| `orchestrator.quality_gate` | `"make check"` | Verify code after every session |
| `models.coding` | `"ADVANCED"` | Best model for code quality |
| `archetypes.skeptic` | `true` | Code review before merge |
| `archetypes.verifier` | `true` | Post-code test verification |
| `archetypes.oracle` | `true` | Spec-drift detection |
| `archetypes.auditor` | `true` | Test quality enforcement |
| `archetypes.instances.verifier` | `2` | Deeper verification coverage |

## Code Default Changes

Where feasible, update the underlying config schema defaults:

- `ArchetypeInstancesConfig.verifier`: `1` -> `2`
- `OrchestratorConfig.quality_gate` promoted in template with `"make check"`
  (code default stays `""` since not all projects use `make`)

## Template Format

The simplified template should:

- Use a clear header explaining the file's purpose
- Group related settings with blank-line separation
- Use `##` comment lines that explain settings in plain language
- End with a footer: `## For all configuration options, see docs/config-reference.md`

## Config Reference Documentation

Create `docs/config-reference.md` with:

- Every config section and field
- Field type, default value, valid range
- Plain-language description of what each field controls
- Examples for complex fields (inline tables, thinking config)
- Organized by section with a table of contents

## Re-initialization

When a user runs `agent-fox init` on an existing project:

- `init` (merge mode) preserves existing user values, adds missing fields
- `init --force` produces the new simplified template
- Both modes produce output consistent with the simplified format

## Non-Goals

- Changing config loading or validation logic
- Modifying the Pydantic config models (except default changes noted above)
- Changing the `hack/` directory configs
- Adding new config fields

## Clarifications

- **Scope**: Changes both the template generator output and underlying code
  defaults where feasible.
- **Hidden sections**: Omitted entirely from template; users learn about them
  from `docs/config-reference.md`.
- **Backward compatibility**: Existing configs with any section continue to load
  and work. The simplification only affects what `init` generates.
- **quality_gate default**: Template shows `"make check"` but the code default
  remains `""` since the setting is project-specific.
